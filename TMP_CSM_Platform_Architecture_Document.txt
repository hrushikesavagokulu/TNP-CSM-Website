1. Project Overview
TMP is the official web platform for the CSM department. It has three access tiers:
1. Public (unauthenticated) — Home page with department info, faculty links, scheme/syllabus links.
2. Student (authenticated) — Personal dashboard with profile, announcements, events, skill roadmap, certifications, company info,
alumni repositories, achievements, learning resources, resume guide, and a community chat ("Connect Sphere").
3. Admin (authenticated, elevated role) — Full control panel to manage every piece of content on the platform, manage students/
admins in bulk, run events, moderate chat, and monitor student data.
The defining design principle across the platform: content modules (skill roadmap, achievements, company info, certifications)
must be schema-flexible, i.e. admins can add content blocks of any type (text, link, video, file, image) in any order, rather than the
frontend enforcing a rigid fixed template. This drives a specific data-modeling decision explained in Section 3.1.

2.5 Data Flow Overview (Request Lifecycle)
Browser → Nginx → Backend (Express)
1. Middleware chain: CORS → Helmet → rate-limiter (Redis) → JSON parse
2. Auth middleware: verify JWT from httpOnly cookie / Authorization header
3. Role middleware: requireRole('student'|'admin') where applicable
4. Controller: validates input (Joi/Zod) → calls service layer
5. Service layer: talks to Mongoose models / Redis / Google Drive / Zoom / Mail
6. Response shaped per-role (e.g., profile fields hidden for peer view)
7. JSON response → Frontend → React Query cache → UI render

2.6 Security Architecture
• Passwords: bcrypt hash (cost factor 12), never returned in any API response.
• JWT: short-lived access token (15 min) in memory/localStorage is avoided — stored in httpOnly, Secure, SameSite=strict cookie;
refresh token similarly, rotated on use.
• OTP: 6-digit numeric, hashed (bcrypt) before storing in Mongo/Redis, TTL 5–10 minutes, max 5 attempts, rate-limited per email
(Redis).
• RBAC: every mutating route explicitly declares required role; profile responses are shaped server-side per viewer role (never trust
frontend to hide fields).
• File upload validation: MIME-type allow-list (images/PDF only for certificates), file-size caps, virus-scan hook placeholder (ClamAV
optional).
• Rate limiting: express-rate-limit + Redis store on /auth/* and /otp/* routes to prevent brute force.
• Helmet.js for HTTP security headers; CORS locked to the frontend origin only.
• All Drive/Zoom credentials stored as Docker secrets / .env , never committed, never exposed to frontend.

• Fixed-field collections - User, Event, EventRegistration, OtpToken, Message - these have a known, stable shape.

instead of a rigid fixed schema that would need a migration every time a new content type is needed.

3.2 API Design Conventions
• Base path: /api/v1/...
• REST resource-oriented routes, JSON in/out.
• Standard response envelope: { success, data, message, error } .
• Pagination: ?page=&limit= on list endpoints (announcements, events, alumni repos, messages).
• Filtering: ?status=&year=&company= query params on relevant list endpoints.
• All admin-mutation routes prefixed /api/v1/admin/... and gated by requireRole('admin') .

3.4 OTP Flow Detail
1. Generate 6-digit numeric code, otpHash = bcrypt.hash(code) .
2. Store {email, otpHash, purpose, expiresAt: now+10min, attempts:0} - Mongo TTL index auto-purges expired docs; Redis
key with EXPIRE used as a fast-path duplicate-send guard (60s cooldown between OTP requests).
3. Verify: fetch latest un-expired doc for {email, purpose} -> bcrypt.compare(inputOtp, otpHash) -> increment attempts on
failure -> block after 5 failed attempts -> delete doc on success.
4. Reused identically for registration, forgot-password, and profile password change.

3.5 File Upload / Google Drive Flow (Event Certificates)
1. Admin creates Event -> backend calls Drive API (service account) -> creates root folder named EventTitle_EventId -> stores
driveFolderId on Event doc.

2. Student submits certificates -> backend uploads directly into the event's Drive folder using naming convention
Name_RollNo_CertificateName.<ext> (one file per certificate).

3. Only image/* and application/pdf MIME types accepted (validated server-side).
4. EventRegistration.uploads[] stores the returned driveFileId + metadata for in-app listing.
5. On event auto-expiry (cron passes expiresAt ), the Event + its EventRegistration documents are deleted from MongoDB, but
the Drive folder and files are retained.

3.6 Zoom Integration Flow
1. Backend authenticates via Zoom Server-to-Server OAuth, caches the short-lived access token in Redis until near-expiry.
2. On registration for a mock-interview event, backend calls Zoom POST /users/{hostEmail}/meetings to create/reuse a
scheduled meeting per the event's config.
3. Returned join_url + start_time saved to EventRegistration .
4. Nodemailer emails the registrant the join link and scheduled time.

3.7 Real-Time Design (Socket.io)
• Namespace /chat : joined per ChatSpace room; events message:new , message:delete , reaction:toggle .
• Namespace /notify : per-user room; event announcement:new pushes live unread-badge updates without polling.
• Redis adapter ( @socket.io/redis-adapter ) attached so this scales horizontally across backend replicas later.

4.1 PUBLIC PAGES

4.1.1 Home Page ( / )
Purpose: Public-facing landing page introducing the department; no authentication required.
UI Layout & Elements (top to bottom): - Navbar: GPREC logo (left) + "CSM Department" wordmark, a few nav items (Home, About links

Edge Cases: If a scheme link section has zero entries, render nothing (no broken empty card). If department-info hasn't been set by

4.1.2 Register Page ( /register )
Purpose: New student account creation with email OTP verification.
UI Layout & Elements: - Centered card form with fields: Name, Roll No, College Gmail, Password, Retype Password, Phone Number,
Branch (dropdown), and a "Send OTP" button next to the email field. - On "Send OTP" click: OTP input field appears (6 boxes), with a

Features: - Real-time validation: password strength, password/retype match, roll number/email uniqueness check (debounced backend
call). - OTP sent to the entered Gmail; backend verifies OTP before account creation is finalized. - Only college Gmail domain accepted

Data Source / API: - POST /auth/check-availability (debounced, checks rollNo/email uniqueness) - POST /auth/register →
sends OTP - POST /auth/verify-otp → creates account, issues JWT, redirects to Dashboard
Access Control: Public route; redirects to Dashboard if already logged in.
Edge Cases: Expired/incorrect OTP shows inline error with remaining attempts; resend is rate-limited (60s cooldown); duplicate roll
number/email blocks submission with a clear message before OTP is even sent.

4.2.3 Announcements Page ( /dashboard/announcements )
Purpose: Year/batch-targeted announcement feed, group-chat-like sidebar experience.
UI Layout & Elements: - Left bar: list of announcement groups the student belongs to — computed from their year and, for years

qualifies for, based on year / batchType fields on their User document.
Data Source / API: - GET /api/v1/student/announcement-groups - GET /api/v1/student/announcements?group= - POST /
api/v1/student/announcements/:id/mark-read - Socket: /notify → announcement:new

Access Control: requireAuth , group membership enforced server-side (a student cannot fetch a group's announcements they don't
qualify for).
Edge Cases: A first-year student only ever sees the single "Year 1" group per requirement (no batch-type sub-groups until year 3+). If
batchType is null for a 3rd/4th-year student (not yet assigned), they see only the plain year-level group until admin assigns a batch type.

4.4 Cross-Cutting UI Behavior: Theme Toggle
• Implemented via a ThemeContext at the app root, backed by a data-theme="light"|"dark" attribute on <html> and CSS
custom properties for every color used across the app (never hardcoded hex values in components).
• Toggle control lives once, in the shared Navbar component (rendered identically on public site, student dashboard, and admin
dashboard), so the same control works "everywhere" per requirement.
• Persisted client-side (localStorage) so it survives refreshes/navigation; default on first visit is light mode.

7. Key User & Data Flows
7.1 Registration Flow
Home -> Register -> fill form -> "Send OTP" -> backend validates uniqueness,
mails OTP -> user enters OTP -> "Register" -> backend verifies OTP, creates User,
issues JWT cookies -> redirect to /dashboard

7.2 Login Flow
Login form -> POST /auth/login -> backend checks password -> issues JWT ->
frontend reads user.role -> role=student => /dashboard, role=admin => /admin

7.3 Forgot Password Flow
Forgot Password -> enter email -> OTP mailed -> enter OTP -> resetToken issued
-> enter new password -> POST /auth/reset-password -> redirect to /login

7.4 Announcement Targeting Flow
Admin composes announcement -> selects target years/batchTypes -> POST /admin/announcements
-> for every currently-online student whose (year,batchType) matches a targetGroup,
backend emits socket 'announcement:new' to their /notify room (live badge update)
-> on next Announcements page load, GET /student/announcement-groups recomputes
membership fresh from the student's current User.year/batchType (always live, not cached)

7.5 Event Registration + Zoom + Drive Flow
Admin creates Event(type=mock-interview) -> backend provisions Drive folder (if cert
uploads allowed) and stores driveFolderId
Student -> Events page -> Register -> backend creates EventRegistration ->

19

if type=mock-interview: backend calls Zoom API -> stores join_url/start_time
-> Nodemailer emails the student the link
Student -> (for certification-drive events) uploads certificate -> backend validates
mime/size -> uploads to Drive under event's folder with Name_RollNo_CertName.ext
-> stores driveFileId in EventRegistration.uploads[]
Event expires (cron) -> Event + all EventRegistration docs deleted from MongoDB;
Drive folder/files persist untouched

7.6 Connect Sphere Message Flow
Student opens Connect Sphere -> socket joins room(spaceId) -> GET recent message
history (paginated) -> sends message -> POST persists to Mongo -> backend emits
'message:new' to room -> all connected clients in that room append the message live
Delete: sender or admin -> DELETE -> backend sets deletedForAll=true -> emits
'message:delete' -> all clients remove it from view (no per-user "delete for me")

7.7 Admin Content Publish Flow (generic, applies to Roadmap/Certs/Achievements/etc.)
Admin opens Manage<Module> page -> ContentBlockEditor -> add/reorder/remove blocks
of any type -> Save -> PATCH/POST to admin.<module> route -> backend persists
contentBlocks array as-is (schema-flexible) -> student-facing page re-fetches and
renders via the shared ContentBlockRenderer component

9. Non-Functional Requirements & Best Practices
• Validation everywhere: every request body validated (Joi/Zod) before touching the DB.
• Consistent error shape: all errors flow through errorHandler.middleware.js into {success:false, message, error} .
• Indexes: unique indexes on email / rollNo ; compound index on {year, batchType} ; TTL index on OtpToken.expiresAt .
• Backups: scheduled mongodump of the mongo-data volume.
• Logging: structured request logging (morgan/winston) in the backend container, shippable to a log aggregator later.
• Testing: Jest + Supertest for backend route/controller tests; React Testing Library for frontend component tests.
• CI/CD ready: each container builds independently, so a CI pipeline can build/push/deploy them separately.
